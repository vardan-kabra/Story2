# Ask HR — Pilot

A small full-stack prototype exploring how an AI agent can **safely** integrate
with an ERP-like database and answer natural-language HR queries for a
multi-campus school system.

> **Core principle:** Claude never generates or executes arbitrary SQL. It may
> only choose from a set of **predefined safe query functions**; the backend
> validates permissions and executes the approved function.

This is a prototype that may later integrate with **Nucleus ERP** (see the
[Future Integration with Nucleus ERP](#future-integration-with-nucleus-erp)
section).

---

## How it works

```
User question ─▶ /api/ask ─▶ Claude (Anthropic API)
                                │   picks a safe tool (never SQL)
                                ▼
                       Backend validates role  ──▶  ❌ denied → told to Claude
                                │  ✅ allowed
                                ▼
                       Safe query layer (parameterised SQLite queries)
                                │  records
                                ▼
                       Claude writes a concise, grounded answer
                                ▼
       API returns { answer, tool used, source records, confidence note }
```

- **Frontend** (`app/page.tsx` + `components/`): single-page chat UI with a text
  input, submit button, answer area, a table of source records, query history,
  a role selector, and loading/error states.
- **Backend** (`app/api/ask/route.ts` + `lib/`): receives the query, runs the
  Claude tool-use loop, enforces access control, executes only allow-listed
  safe functions, and returns the answer plus provenance.

## Setup

### Prerequisites

- Node.js 18.18+ (Node 20/22 recommended)
- An Anthropic API key

> No C/C++ build tools required — the SQLite engine is **sql.js** (pure
> WebAssembly), so `npm install` works on Windows/macOS/Linux without a compiler.

### 1. Install dependencies

```bash
cd ask-hr-pilot
npm install
```

### 2. Create `.env.local` and add your API key

Copy the example file and fill in your key:

```bash
cp .env.example .env.local
```

Then edit `.env.local`:

```dotenv
ANTHROPIC_API_KEY=sk-ant-your-real-key-here
```

The key is read from the `ANTHROPIC_API_KEY` environment variable — **no secrets
are hardcoded**, and `.env.local` is gitignored.

Optional overrides:

- `ASK_HR_MODEL` — Claude model id (default `claude-opus-4-8`).
- `ASK_HR_DB_PATH` — SQLite file path (default `./data/ask-hr.db`).

### 3. Seed the database

```bash
npm run seed
```

This creates `data/ask-hr.db` from `data/seed.sql` (campuses, employees, leave
balances, and HR requests). Re-running it rebuilds the database from scratch.

### 4. Run the app

```bash
npm run dev
```

Open <http://localhost:3000> and start asking questions.

For a production build:

```bash
npm run build && npm start
```

## Safe query layer

The only ways Claude can touch data are these allow-listed functions
(`lib/safeQueries.ts`, exposed as tools in `lib/tools.ts`):

- `getEmployeeByName(name)`
- `getLeaveBalance(employeeName)`
- `getEmployeesByCampus(campus, filters?)`
- `getEmployeesByDepartment(department)`
- `getPendingHRRequests(days)` — 🔒 elevated
- `getPolicyByTopic(topic)`
- `getReportingManager(employeeName)`
- `getDirectReports(managerName)`
- `summarizeLeaveStatus(campus, role?)` — 🔒 elevated

Every function uses **parameterised** SQL (user input is never interpolated into
SQL text), and the database is loaded **read-only** (in-memory via sql.js) and
never written back by the app.

### Policy handling

HR policies live in a small JSON knowledge base (`data/hrPolicies.json`). Policy
questions are answered via `getPolicyByTopic`, which does lightweight
keyword/title matching and returns each entry **with its `source` citation**.
The agent is instructed to always cite the policy source, and to clearly say
when no matching policy is found.

## Security design notes

- **No arbitrary SQL.** Claude can only select from predefined tools. It cannot
  write or run SQL. All queries are parameterised and the DB is read-only.
- **Allow-listed tool surface.** The backend executes a function only if it
  exists in the `TOOLS` registry (`lib/tools.ts`).
- **Role-based access control.** `lib/auth.ts` gates sensitive tools. Employee-
  wide summaries (`summarizeLeaveStatus`) and pending HR requests
  (`getPendingHRRequests`) require `HR_ADMIN` or `CAMPUS_HEAD`. Less-sensitive
  data (leave balances, reporting managers, general policies) is visible to all
  roles. Permission is checked **before** any function executes; denied calls
  are reported back to Claude, which explains the restriction rather than
  working around it.
- **No hardcoded secrets.** The API key is read from `ANTHROPIC_API_KEY`.
- **Grounded answers + provenance.** The API returns the tool(s) used, the
  source records, the data sources/policy citations, and a confidence/uncertainty
  note. The model is instructed never to fabricate data.
- **Prototype caveats.** The role is supplied by the client for demo purposes,
  there is no authentication, and the data is synthetic. These are intentionally
  simplified — see below for how this hardens in production.

## Future Integration with Nucleus ERP

In a production deployment integrated with **Nucleus ERP**:

- **Data source.** The local SQLite database would be replaced by **Nucleus ERP
  APIs or read replicas**. `lib/db.ts` / `lib/safeQueries.ts` become the
  integration seam.
- **Safe query mapping.** Each safe query function would map to an **approved
  ERP endpoint** (or a vetted, parameterised read query against a replica)
  rather than local SQL — preserving the "no arbitrary SQL" guarantee.
- **Real permissions.** Access control would use **real user roles from
  Nucleus** (and campus scoping), resolved from an authenticated session/JWT,
  instead of the client-supplied role placeholder in `lib/auth.ts`.
- **Audit logging.** Every query and data access would be written to an **audit
  log** (who asked what, which tool ran, which records were returned).
- **Stricter sensitive-data controls.** Sensitive HR data (compensation,
  performance, medical/leave details) would require **stricter, field-level
  access control** and possibly redaction.
- **Approved policy documents.** Policy answers would be sourced from
  **approved, version-controlled HR policy documents** instead of the sample
  JSON knowledge base, retaining mandatory source citation.

The code marks these integration points with `NUCLEUS ERP INTEGRATION POINT`
comments (see `lib/auth.ts`).
